# ADR-003: Evidence Capture Strategy — Terminal Output Only

**Status:** ACCEPTED  
**Date:** 2026-06-27  
**Author:** ARCHITECT  
**Approved by:** Human on 2026-06-27

---

## Context

ASES requires evidence to prevent false confidence claims. "This works" is not evidence. But what counts as evidence?

Claude can run commands (tsc, jest, eslint, gradle) and see the output. Claude can also read files. But Claude generates both — it produces the command output and reads it. How do we know the evidence is real?

We need to distinguish:
- **Real evidence:** Terminal output captured to a file (tool ran, output is permanent, human can verify)
- **Claimed evidence:** Claude's assessment of its own work ("I checked it and it works")

---

## Problem Statement

We need an evidence system that:

1. **Is verifiable by humans** — Human can open the file and see the actual output, not Claude's summary
2. **Is not fabricatable by Claude** — Claude cannot generate fake evidence
3. **Is timestamped** — When did this verification run?
4. **Is persistent** — Evidence stays on disk, not just in Claude's memory
5. **Covers all stacks** — TypeScript, Python, Kotlin/Java — each has different tools
6. **Has clear hierarchy** — Some evidence is stronger than others

---

## Alternatives Considered

### Option A: Claude's Confidence Assessment (rejected)
**Description:** Claude reads code and assesses correctness. If confident, claim it's verified.

**Pros:**
- Simple to implement
- Fast (no actual verification)

**Cons:**
- This is exactly what leads to fabricated confidence
- No actual evidence
- Unverifiable by humans
- Leads to the core failure mode we're trying to prevent

**Rejection Reason:** Defeats the entire purpose of evidence gates.

### Option B: Log Files Generated by Tools (chosen) ✓
**Description:** Tools (compiler, linter, tests) produce output. Capture to timestamped files. Claude interprets logs but never generates them.

**Pros:**
- ✓ Verifiable by humans (file is on disk, can be opened independently)
- ✓ Not fabricatable by Claude (tool produces output, Claude just captures it)
- ✓ Timestamped (filename includes epoch timestamp)
- ✓ Persistent (files stay on disk, not ephemeral)
- ✓ Covers all stacks (each tool has different output format, but all are captured)
- ✓ Trusted (tool output is authoritative, not Claude's opinion)

**Cons:**
- ✗ Claude must actually run commands (not cached or guessed)
- ✗ Environment must be set up (tools must be installed)
- ✗ Cannot be faked (but that's a pro, not a con)

**Selection Reason:** This is the only approach that prevents fabricated evidence. Evidence must be generated by tools, captured by Claude, interpreted by Claude, verified by humans.

### Option C: Hybrid (human + tool evidence) (rejected)
**Description:** Accept both tool output and human spot-checks.

**Pros:**
- Flexible (allows both automated and manual verification)

**Cons:**
- Introduces trust issues (human spot-checks can be wrong)
- Harder to audit (mix of evidence types)
- Still doesn't prevent tool-less environments (e.g., no jest, no tsc)

**Rejection Reason:** Adding manual evidence to tool output introduces more problems. Better to standardize on tools.

## Decision

Evidence is defined as:

### Evidence Sources (Ranked by Trust)

**Level 1 — STRONGEST: CI/CD System Output**
- GitHub Actions logs, GitLab CI logs, Jenkins builds
- Runs in isolated environment, not influenced by local state
- Cannot be influenced by developer
- Example: GitHub Actions pytest output

**Level 2 — STRONG: Human-Provided Terminal Output**
- Human runs command in terminal, captures output to file
- Human can vouch for the execution
- Cannot be faked (human was present)
- Example: User runs `pytest` and saves output to `.ases/evidence/task-001/test.log`

**Level 3 — ACCEPTABLE: Claude CLI Executing Commands**
- Claude runs command in Claude CLI environment
- Captures output to timestamped file
- Output is verifiable by opening the file
- Weaker than Level 1 (Claude could potentially lie) but stronger than no evidence
- Example: `jest --coverage 2>&1 | tee .ases/evidence/task-001/test-1719534330.log`

**Level 4 — NOT ACCEPTED: Claude Assessment**
- Claude reads code and assesses correctness
- No actual tool output
- Completely unverifiable
- Fabricatable by Claude
- Example: "I checked the code and it looks correct" — NOT EVIDENCE

### Evidence Capture Format

All evidence log files follow this format:

```
[command output here]
[stdout and stderr mixed]
[all text from the tool]

EXIT: [exit code — 0 for success, non-zero for failure]
TIMESTAMP: [2026-06-27T23:55:30Z]
```

### Stack-Specific Evidence

#### TypeScript/Ortho
Tools: tsc, eslint, jest, npm

```bash
# Type checking
tsc --noEmit 2>&1 | tee .ases/evidence/task-001/build-$(date +%s).log
echo "EXIT: $?" >> .ases/evidence/task-001/build-$(date +%s).log
echo "TIMESTAMP: $(date -u +'%Y-%m-%dT%H:%M:%SZ')" >> ...

# Linting
eslint . 2>&1 | tee .ases/evidence/task-001/lint-$(date +%s).log

# Tests
jest --coverage 2>&1 | tee .ases/evidence/task-001/test-$(date +%s).log
```

#### Python/Ortho
Tools: ruff, mypy, pytest

```bash
# Linting
ruff check . 2>&1 | tee .ases/evidence/task-001/lint-$(date +%s).log

# Type checking
mypy . --strict 2>&1 | tee .ases/evidence/task-001/types-$(date +%s).log

# Tests
pytest --tb=short --cov=. 2>&1 | tee .ases/evidence/task-001/test-$(date +%s).log
```

#### Kotlin/Java/Expense App
Tools: gradle, ktlint

```bash
# Linting
./gradlew ktlintCheck 2>&1 | tee .ases/evidence/task-001/lint-$(date +%s).log

# Build
./gradlew build 2>&1 | tee .ases/evidence/task-001/build-$(date +%s).log

# Tests
./gradlew test 2>&1 | tee .ases/evidence/task-001/test-$(date +%s).log
```

### Evidence Declaration

Every verification report must declare its source:

```
EVIDENCE-SOURCE: CLAUDE-CLI | HUMAN-TERMINAL | CI-CD | MIXED
```

If CLAUDE-CLI: Claude ran commands, captured logs  
If HUMAN-TERMINAL: Human provided logs from manual terminal run  
If CI-CD: Output from GitHub Actions or similar  
If MIXED: Different checks from different sources (note which is which)

### What Claude Can and Cannot Do

**Claude CAN:**
- ✓ Run terminal commands
- ✓ Capture output to files
- ✓ Read log files produced by tools
- ✓ Interpret and summarize log file contents
- ✓ Compare tool output to expected results
- ✓ Claim success only if log files show success (exit code 0, tests passed, etc.)

**Claude CANNOT:**
- ✗ Claim to have run a command if it didn't actually run
- ✗ Fabricate log files
- ✗ Summarize evidence from memory (must read actual files)
- ✗ Hide or selectively report log file contents
- ✗ Claim success if log file shows failure

### Evidence Integrity Rules

1. **All-or-nothing:** If one evidence check fails, the gate fails (no cherry-picking passing tests and hiding failing ones)
2. **Verbatim:** When reporting errors, quote exact text from log files (no paraphrasing)
3. **Timestamped:** Every log file must have a timestamp (proves when it ran)
4. **Immutable:** Log files are never edited after creation (append-only, or new files for new runs)
5. **Linked:** Verification reports reference specific log files (human can find and open them)

---

## Rationale

**Why tools, not Claude assessment?**

Claude can produce false confidence. Tools cannot lie. `tsc` either compiles or it doesn't. `jest` either has passing tests or it doesn't. Tool output is objective.

**Why must humans verify?**

Claude could theoretically trick us. By requiring humans to spot-check log files, we prevent mass fabrication. One human checking one log file breaks the entire lie.

**Why three levels of evidence?**

CI/CD > human terminal > Claude CLI
- CI/CD is strongest (isolated environment, no influence from local state)
- Human terminal is strong (human vouches)
- Claude CLI is acceptable but weaker (Claude could theoretically lie, though we try to prevent it)

We accept Claude CLI evidence but document that it's weaker than CI/CD.

**Why no Claude assessment?**

Because this is what leads to fabricated confidence. We're trying to prevent exactly this problem.

---

## Consequences

### Positive
- ✓ Evidence is verifiable by humans
- ✓ Evidence cannot be fabricated by Claude (tools produce it)
- ✓ Evidence is timestamped and persistent
- ✓ Evidence hierarchy is clear (CI/CD > human > Claude)
- ✓ All stacks can produce evidence (tools exist for all)
- ✓ Evidence integration is clear (gates require evidence)

### Negative
- ✗ Requires actual tool execution (not cached or guessed)
- ✗ Requires tools to be installed (dependencies)
- ✗ Slower than assuming correctness
- ✗ Cannot verify things without tools (design decisions, architecture, subjective quality)

### Neutral
- ~ Log files on disk are persistent (good for auditing, takes disk space)
- ~ Timestamping means no cache reuse (every run is fresh, slower but accurate)

---

## Future Considerations

1. **If tool execution is impossible:**
   - Declare EVIDENCE-SOURCE: HUMAN-TERMINAL
   - Wait for human to provide logs from their terminal
   - Do not fabricate tool output

2. **If tools are missing:**
   - Document in log file that tool is not installed
   - Do not fake tool output
   - Flag as blocker (cannot verify without tool)

3. **If tool output is ambiguous:**
   - Quote the exact output
   - Do not interpret or smooth-talk it
   - Flag for human review

4. **If evidence is insufficient:**
   - Do not pretend (no "this should work" claims)
   - Document what cannot be verified
   - Ask for manual verification
